Apex Systems Deployment Guide
Revision 2.1 — January 2026

This guide is the authoritative reference for shipping code to any environment operated by Apex Systems. It applies to every engineer who merges code to the monorepo, regardless of seniority or team. Deviations from this process require written approval from the VP of Engineering and are documented in the exceptions log.

Environments

Apex operates four environments with distinct purposes, access controls, and deployment rules.

Local is the development environment that every engineer runs on their workstation. It is defined as a docker-compose profile in the monorepo root. Local spins up all services, an in-process Kafka broker, a PostgreSQL instance, a MinIO S3-compatible object store, and a lightweight Trino container. Engineers are expected to keep local fully functional; a broken local environment that is not fixed within 24 hours should be escalated to the DevEx team. Seed data scripts are included for each service so that engineers do not interact with staging data.

Staging is a full Kubernetes environment hosted in us-east-1 that mirrors the production topology, including all three worker node groups, the full observability stack, and real SSL certificates. Staging serves synthetic data generated by a daily data generator job. Engineers use staging for integration testing, external demos, and validating infrastructure changes. Staging is shared across the team; destructive tests that affect shared state must be coordinated in the #staging-coordination Slack channel.

Canary is a 5% traffic slice of the production cluster. A separate Kubernetes service revision is deployed, and traffic is split using Argo Rollouts with a weighted VirtualService in Istio. Canary exists to validate behavioral changes under real customer traffic before full promotion. Engineers may observe canary metrics in the Grafana canary dashboard, which is automatically pinned to the revision deployed.

Production serves 100% of customer traffic across three regions. Deployment to production is fully automated after a successful canary period. Engineers do not deploy to production manually; the delivery pipeline handles promotion.

Source Control and Code Review

All services, libraries, infrastructure definitions, Helm charts, and CI configuration live in a single monorepo hosted on GitHub. The monorepo uses Bazel for builds with a shared build cache hosted on a dedicated S3 bucket. Bazel ensures that only affected targets are rebuilt and retested, which keeps CI times bounded even as the repository grows.

Developers branch from main using the naming convention owner-username/APEX-ticket-number-short-description. Branches older than 30 days without an open pull request are automatically deleted by a nightly GitHub Actions workflow.

Direct pushes to main are disallowed by branch protection rules. Every change requires a pull request. Pull requests require: at least one approval from a person listed in the CODEOWNERS file for the changed path, passing required status checks, no unresolved review comments marked as blocking, and no merge conflicts. Force-pushing to open PRs is disallowed. Approved PRs are merged using the squash-and-merge strategy to keep the main branch history linear.

For changes that affect the API contract, the Helm values schema, the Kafka message schema, or the database migration files, a second approval from the platform-owners team is required in addition to the CODEOWNERS approval. This double-approval gate catches backward-incompatibility issues before they reach staging.

Continuous Integration Pipeline

CI runs on GitHub Actions using self-hosted runners on c6i.4xlarge EC2 instances provisioned by the runner manager. Self-hosted runners avoid the memory and disk limits of GitHub-hosted runners and allow Bazel's remote cache to be used without network latency.

The standard pipeline is triggered on every push to a pull request branch and on every merge to main. The pipeline stages run in order:

Stage 1 — Build: Bazel builds all affected targets. The build cache hit rate is monitored; a cache miss rate above 15% triggers a review of the cache key configuration.

Stage 2 — Unit tests: Bazel runs all unit tests affected by the changed targets. Tests must complete in under 5 minutes per service; tests that exceed this threshold are flagged for splitting or optimization.

Stage 3 — Integration tests: Integration tests run against ephemeral Kind clusters provisioned by the test framework. Each Kind cluster is bootstrapped with the current chart versions and the changed service revision. Integration tests cover the happy path and documented error paths for each service API.

Stage 4 — Static analysis: golangci-lint for Go services, ruff for Python, clippy for Rust, eslint for TypeScript. Analysis failures are treated as build failures.

Stage 5 — Security scans: Semgrep runs a ruleset covering OWASP Top 10 and Apex custom rules for common misconfigurations. Trivy scans every container image for known CVEs with critical or high severity; images with unfixed critical CVEs are blocked from promotion. The SBOM is generated for every image and stored in the internal artifact registry.

Total pipeline duration must remain under 25 minutes. The DevEx team monitors pipeline duration weekly and works with teams to optimize slow stages.

Continuous Delivery and Progressive Delivery

Merges to main trigger the delivery pipeline via Argo CD. The pipeline is defined as an Argo Workflow that orchestrates build, sign, publish, and deploy stages.

Builds are triggered by a push event from GitHub. The container image is built by Kaniko running inside the cluster, avoiding the need for Docker daemon access. The image is signed with cosign using a key stored in AWS KMS. The signed image is pushed to the internal ECR registry. Image signing is verified at deployment time by Kyverno policies; images without a valid signature are rejected by the admission controller.

Deployment to staging is fully automatic on every merge to main. The Helm chart for the changed service is updated with the new image tag, committed to the gitops repository, and synced by Argo CD within two minutes.

Deployment to canary is gated by a human approval step in the Argo Workflow. The engineer who merged the change, or the on-call engineer, clicks Approve in the Argo UI after verifying that staging tests have passed. Canary deployments are permitted only during business hours (09:00 to 17:00 Pacific Monday through Friday) unless the change is a rollback of an active incident. The canary revision receives 5% of production traffic, held for 30 minutes.

Promotion from canary to production is automatic if no SLO-based alerts have fired during the canary window. Promotion triggers a blue-green deployment in each of the three production regions, sequenced east-to-west with a 5-minute wait between regions. If an SLO alert fires at any point during the canary window or during production rollout, the delivery pipeline halts and pages the on-call engineer.

Feature Flags

All changes that modify user-visible behavior, change a data schema, or affect customer-facing performance characteristics must be gated behind a LaunchDarkly feature flag before merging. Flags default to off in all environments. The flag name must match the Jira ticket number using the format APEX-{number}-{short-description}.

Flags are enabled progressively: first for internal Apex accounts, then for a 1% cohort of tenants, then 10%, then 50%, then 100%. Each cohort expansion is a manual step in LaunchDarkly performed by the feature owner after verifying that error rates, latency, and business metrics are nominal at the current cohort size.

A flag that reaches 100% rollout and has been stable for 30 days must be removed from the codebase. The DevEx team publishes a weekly report of flags eligible for removal. Flags that are not removed within 60 days of eligibility are tracked as technical debt in the reliability review.

Secrets and Configuration Management

Secrets — API keys, database passwords, signing keys, OAuth credentials — are stored in AWS Secrets Manager. The External Secrets Operator watches Kubernetes CRDs that reference Secrets Manager paths and materializes the secret values as Kubernetes Secret objects at pod startup. Secrets are never printed in logs; log sanitization middleware strips values matching known patterns.

Application configuration — feature defaults, resource limits, log levels, timeouts — uses Helm values organized in a layered structure: base values defined in the chart, environment-specific overlays committed to the gitops repository, and per-tenant dynamic configuration served by the control plane configuration API at runtime. Engineers should prefer the control plane API for any configuration that must change frequently or differ by tenant.

Secrets and configuration are never baked into container images. Any image layer that contains a secret is considered compromised and triggers a rotation incident. An accidental commit of a plaintext secret to the repository triggers an automated response: the secret is flagged by the GitHub secret scanning integration, the rotation runbook is triggered, and an incident is created in PagerDuty.

Rollback Procedures

Every deployment records an immutable Helm release history in the cluster. Argo CD maintains the complete sync history for each application. A rollback in Argo CD reverts to the previous stable revision within approximately 60 seconds and is the preferred remediation for any regression introduced by a new deployment.

Database migrations are managed by golang-migrate. Migrations must be backward compatible with the previous application revision. This means that before removing a column, the application must stop reading from it; before renaming a column, the application must support both names. A migration that is not backward compatible must be staged across multiple releases with explicit coordination between the migration author and the on-call team.

If a rollback cannot resolve an incident because the migration is already applied and the previous revision does not support the new schema, the incident commander convenes the relevant service owners to write a forward-fix migration and an application patch. This scenario is considered a process failure and is included in the postmortem.

On-Call Operations

Each service domain has a named primary and secondary on-call engineer in PagerDuty. Rotations are weekly, starting Monday at 09:00 Pacific. Engineers must acknowledge a PagerDuty alert within 5 minutes. Failure to acknowledge triggers escalation to the secondary. Failure of the secondary to acknowledge within an additional 5 minutes escalates to the on-call engineering manager.

Severity 1: customer-impacting service degradation or outage. Requires an incident commander, a dedicated Slack incident channel, and a live status page update within 15 minutes. Resolution is the only priority.

Severity 2: degraded service that is not causing direct customer impact but poses a risk of escalation. Addressed during business hours unless the risk assessment changes.

Severity 3: non-customer-impacting internal alerts, performance regressions, or elevated error rates within acceptable bounds. Tracked in Jira and addressed in the next sprint.

On-call engineers are compensated with a $200 per week on-call stipend regardless of whether any incidents occur. Incidents that require significant out-of-hours work — defined as more than two hours outside of normal working hours in a single week — are compensated with one additional day of PTO.

Incident Management and Postmortems

All severity 1 and severity 2 incidents require a blameless postmortem document. The postmortem must be published in the internal wiki within five business days of incident resolution. The template requires: a concise incident summary, an impact statement with quantified customer and business effect, a detailed timeline, a root cause analysis, a list of contributing factors, and a set of action items with named owners and target dates.

Action items are created as Jira tickets with the label postmortem-action. They appear in the monthly reliability review dashboard. Action items that are not resolved within 30 days are escalated to the engineering manager for the responsible team.

Change Freeze

Apex observes a mandatory change freeze from December 20 through January 2 each year to protect customers during the end-of-year period. Additionally, change freezes may be declared by the VP of Engineering for periods surrounding major customer events such as contract renewals, audit windows, or penetration tests.

During a change freeze, only rollbacks and security-critical changes may be deployed. Security-critical changes are those that address a confirmed actively exploited vulnerability or correct a data exposure. Each such deployment requires written approval from the VP of Engineering, documented in the exceptions log with a severity justification and a description of the change.
